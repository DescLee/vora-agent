# 项目检索效率优化设计

## 目标

减少项目问答中的无效目录扫描、重复文件读取和 ReAct 往返轮次，使常见问题能够通过一次候选定位和一次并行证据读取完成回答。

## 范围

本次按三个阶段实施：

1. 小文件 `read_file` 直接向模型内联正文。
2. 向项目只读任务开放 `glob`，并增加结构化 `search_code` 工具。
3. 建立通用、增量缓存的项目索引，为 Planner 和 ReAct 提供候选文件与符号提示。

不引入向量数据库、embedding 服务或新的外部依赖。

## 设计

### 小文件读取

- 普通 `read_file` 结果在正文不超过现有内联上限时直接进入模型上下文。
- 超过上限时继续使用 `content_ref`，要求模型通过 `query` 或行号窗口定向读取。
- 定向读取保持现有行为，避免改变大文件保护策略。

### 定位工具

- `glob` 加入普通只读、代码审查、代码验证和文件输出任务的默认工具集合。
- 新增 `search_code` 只读工具，使用 Python 标准库遍历文本文件并执行正则或纯文本搜索。
- 返回结果包含 `path`、`line` 和 `snippet`，并限制文件数、命中数、文件大小和输出长度。
- 搜索遵循工作区边界、敏感文件保护和 `.gitignore` 规则。

### 项目索引

- 新增独立的项目索引模块，不继续扩张 `context.py`。
- 索引内容包括项目类型、入口候选、manifest、测试目录、源码文件和常见声明符号。
- 使用文件路径、大小和修改时间生成签名，缓存到现有项目缓存目录。
- 索引只提供候选提示，不替代文件原文读取。
- Planner 始终获得精简索引；ReAct 在项目或代码相关任务中获得同一索引，避免重复目录探测。

## 兼容和失败处理

- 文件不可读、解码失败或缓存损坏时跳过单个文件并继续生成索引。
- 非 Git 项目同样可用；存在 `.gitignore` 时沿用现有忽略规则。
- 大型仓库通过文件数量、单文件大小和符号数量上限控制成本。
- 旧的项目结构摘要继续保留，索引作为更通用的补充，后续可在数据验证后替换旧摘要。

## 验证标准

- 小文件普通读取一次即可让模型获得正文。
- `glob` 出现在项目只读任务的工具 schema 中。
- `search_code` 能返回受限、结构化的文件行号和片段。
- 修改源码文件后索引缓存自动失效并更新。
- 现有文件读取、大文件保护、工具边界和上下文测试不回归。
