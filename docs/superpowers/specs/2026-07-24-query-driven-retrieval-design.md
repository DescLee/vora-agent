# 查询驱动项目检索设计

## 目标

将项目问答从“每轮预扫描仓库”改为“围绕用户问题获取最小充分证据”，减少文件遍历、源码读取和 ReAct 往返轮次。

## 核心策略

1. Planner 和 ReAct 只注入轻量 Repo Map，包括项目类型、manifest、顶层目录、常见入口和最近修改文件，不提取全仓符号。
2. `search_code` 一次接受最多三组查询，优先使用 `rg --json` 获取结构化路径、行号和片段。
3. 搜索结果按明确路径、完整符号、文件名和普通文本命中排序。
4. 首轮最多读取六个候选文件；证据不足时只允许再搜索一次，并沿直接调用方、被调用方或测试扩展一层。
5. 修改任务续跑复用本轮已修改文件、既有观察和验证结果，不因单文件变化重建全仓索引。

## 边界

- 不引入向量数据库、embedding 或 Tree-sitter 依赖。
- `rg` 不可用时回退现有 Python 文本搜索。
- 搜索继续遵守工作区、`.gitignore`、敏感文件、文件大小和结果数量限制。
- Repo Map 不替代原文证据，最终结论必须来自 `search_code` 命中或 `read_file` 正文。

## 验收

- 构建 Planner/ReAct 提示词不再读取源码正文或遍历全仓提取符号。
- 单次 `search_code` 可执行多查询并返回查询归属。
- 研究和代码审查任务最多执行两次代码搜索、读取六个不同文件。
- 超出预算时返回可解释 observation，引导模型基于已有证据收口。
