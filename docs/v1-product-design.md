# Manus Mini V1 产品设计文档

## 1. 项目定位

Manus Mini 是一个面向面试展示的 TUI AI Agent 项目。第一版不做 Web 页面，但需要提供可连续对话的终端界面：用户可以先提出一个目标，例如写调研报告；Agent 生成初稿后，用户可以继续追问、要求补充、删改某些段落或调整输出格式。

第一版重点展示一个可运行、可观察、可迭代的 Agent 核心流程：系统围绕会话持续接收用户意图，规划任务、调用工具、观察结果、修正计划，并维护当前产物状态。

第一版覆盖三类代表性能力：

1. 资料调研：围绕主题读取资料、提炼要点、生成 Markdown 报告。
2. 本地项目助手：扫描项目、读取文件、生成修改方案，并可在用户确认后修改文件。
3. 任务自动化：整理本地文本/文件，生成摘要、清单或结构化结果。

项目的核心价值不在于堆功能，而在于展示 Agent 工程能力：三层 Loop Engineering、工具抽象、状态机、执行日志、权限确认、失败恢复雏形。

## 2. 目标用户与使用场景

目标用户是面试官、技术负责人或项目评审者。他们希望快速判断候选人是否理解现代 AI Agent 应用的工程落地方式，而不是只会调用一次 LLM API。

典型演示场景：

```text
用户: 阅读 docs 目录，写一份项目背景调研报告
Agent: 已生成报告初稿，并列出资料来源和待确认问题

用户: 第二部分太泛了，补充技术架构风险
Agent: 已重新读取相关资料，并更新报告第二部分

用户: 把结论改成适合面试讲解的版本
Agent: 已重写结论，并保留原始版本到运行日志
```

面试展示时应突出：

- Agent 如何把自然语言目标拆成步骤。
- ReAct 如何在 LLM 思考和工具调用之间循环。
- 多个工具调用如何判断依赖关系，可并行则并行，有依赖才串行。
- Reflection 如何评判结果并决定局部更新、重新生成或重新规划；代码类任务会额外执行 pytest 验收门禁。
- 工程兜底循环如何处理超时、重试、token 成本、工具异常和最大迭代次数。
- 工具调用如何被约束、记录和审计。
- 文件修改为什么需要确认。
- 执行失败后系统如何观察错误并调整下一步。
- 第一版如何自然演进到第二版。

## 3. 第一版范围

### 3.1 必须完成

- TUI 入口：提供可连续对话的终端页面，支持输入区、消息区、状态区和产物区。
- 三层 Agent Loop：ReAct 工具循环、Reflection 质量反馈循环、工程兜底重试循环。
- 工具并行调度：同一轮多个工具调用先分析依赖关系，无依赖只读工具并行执行，有依赖或写入冲突的工具串行执行。
- 会话状态：保存历史消息、当前任务、当前产物、工具观察和待确认动作。
- 长期记忆：保存跨会话可复用的信息，例如用户偏好、项目摘要、常用输出格式、已生成产物摘要。
- 上下文压缩：当会话变长时，将旧消息、工具观察和产物版本压缩成摘要，保留关键事实和决策。
- 三类工具包：
  - Research Pack：读取本地资料、摘要、生成报告。
  - Code Pack：扫描目录、读取文件、生成补丁或修改文件。
  - Automation Pack：文本整理、文件归类、清单生成。
- 文件写入确认：所有真实文件修改前必须展示变更摘要并等待用户确认。
- 执行日志：记录每一步计划、工具调用、工具结果、反思结论和最终输出。
- Markdown 产物：报告、清单或修改结果默认写入 `outputs/`，并支持后续对话继续修改。
- 基础测试：覆盖 Loop 状态流转、工具协议、文件写入确认逻辑。

### 3.2 暂不完成

- Web 页面。
- 多 Agent 协作。
- 云端任务队列。
- 复杂联网搜索。
- 自动安装依赖或静默执行高风险命令。当前命令工具仅在风险判断和用户确认后执行。
- 对任意仓库的大规模自动重构。

## 4. TUI 交互设计

### 4.1 启动方式

```bash
manus-mini
```

可选参数：

```bash
--cwd <path>             指定工作目录，默认当前目录
--max-steps <number>     外层工程兜底循环最大步数，默认 8
--max-react <number>     单个步骤内 ReAct 工具循环最大次数，默认 5
--max-reflect <number>   Reflection 质量反馈循环最大次数，默认 3
--max-tool-retries <n>   单个工具调用最大重试次数，默认 2
--mode <auto|confirm>    执行模式，默认 confirm
--session <id>           恢复指定会话，第一版可先预留
--dry-run                只展示计划和拟执行动作，不修改文件
```

### 4.2 页面布局

TUI 第一版建议分为四块：

```text
+------------------------------------------------------------+
| Manus Mini                          cwd: ./project         |
+-------------------------+----------------------------------+
| 对话区                  | 当前产物                         |
| 用户/Agent 消息         | report.md / todo.md / patch 预览 |
| 工具调用摘要            |                                  |
+-------------------------+----------------------------------+
| 状态栏: step 3/8 | react 2/5 | reflect 1/3 | waiting      |
+------------------------------------------------------------+
| 输入区: 继续输入你的要求...                                |
+------------------------------------------------------------+
```

第一版不追求复杂视觉效果，重点是让用户能在同一个页面里看到：

- 当前对话上下文。
- Agent 正在做什么。
- 当前产物长什么样。
- 哪些动作需要确认。

### 4.3 运行中的信息展示

TUI 按步骤展示关键事件：

```text
[Plan] 生成 4 个步骤
[Act] 调用 list_files 工具，目标: ./src
[Observe] 找到 12 个文件
[Reflect] 需要读取 package.json 判断项目类型
[Act] 调用 read_file 工具，目标: package.json
```

### 4.4 文件修改确认

当工具准备修改文件时，必须显示：

```text
即将修改: src/config.ts
变更摘要: 新增默认配置导出

是否继续？[y/N]
```

用户选择 `y` 才执行写入。其他输入默认取消该工具调用，并把取消结果交给 Agent 继续反思。

### 4.5 多轮修改

用户的后续输入默认基于当前会话和当前产物继续工作，不需要重新描述背景。

示例：

```text
用户: 阅读 docs 生成调研报告
Agent: 已生成 outputs/research-report.md

用户: 加一段竞品分析
Agent: 已补充竞品分析章节

用户: 删除太主观的表达
Agent: 已重写第 2、4 节，并说明修改点
```

### 4.6 记忆与上下文压缩提示

TUI 不需要暴露复杂记忆管理页面，但需要让用户知道系统正在使用哪些上下文：

```text
状态栏: context 68% | memory: project-summary, writing-style | step 2/8
```

当系统压缩上下文时，在对话区追加一条系统消息：

```text
[System] 已压缩较早的 12 条消息，保留目标、关键结论、文件修改记录和待办事项。
```

压缩必须保证工具调用记录完整：如果某轮工具调用被保留，对应的工具结果也必须保留；如果删除，则整组工具调用和结果一起删除，并用摘要记录结论。

第一版记忆规则：

- 用户明确表达的偏好可以写入长期记忆，例如“报告用面试讲解风格”。
- 项目级摘要可以写入长期记忆，例如目录结构、技术栈、重要文件。
- 敏感内容、临时错误、未确认猜测默认不写入长期记忆。
- 用户可以输入“忘记刚才的偏好”触发删除对应记忆。

## 5. 核心流程

### 5.1 成功路径

1. 用户在 TUI 输入首轮目标。
2. Planner 根据目标生成初始计划。
3. ReAct Loop 执行当前步骤：LLM 决定是否需要工具，Agent 对多个工具调用做依赖分析，可并行则并行执行，再返回观察结果，直到得到草稿结果。
4. Reflection Loop 评判草稿结果：非代码任务当前版本直接放过；代码类任务会生成并执行临时 pytest 验收 case，失败时把原始输入、case 和失败原因回流给下一轮执行。
5. Engineering Loop 检查运行边界：最大迭代次数、token 预算、工具重试和异常兜底。
6. Render 将 Agent 消息、工具摘要和当前产物刷新到 TUI。
7. Memory Manager 提取可长期复用的信息，Context Compressor 在必要时压缩旧上下文。
8. 用户继续输入修改要求，系统基于同一个 SessionState 和相关长期记忆开启下一轮。

### 5.2 失败路径

第一版只处理核心失败：

- 文件不存在：返回明确错误，Agent 可选择列目录或询问用户。
- 工具参数非法：中止当前工具调用，要求重新规划。
- 用户拒绝写入：记录为取消，不视为系统错误。
- 达到最大步骤数：输出已完成内容、未完成事项和下一步建议。
- ReAct 工具循环达到上限：停止继续调用工具，进入 Reflection 判断当前部分结果是否可用。
- Reflection 多次不通过：保留当前最佳结果，并说明质量风险和建议人工介入点。
- 外层工程兜底循环达到上限：输出当前部分结果、未完成事项和建议下一步。
- token 成本过高：先压缩上下文，仍超限则裁剪低优先级上下文。
- 工具超时或异常：按策略重试，重试耗尽后降级或跳过该工具。
- 并行工具批次部分失败：保留成功工具的观察结果，失败工具进入重试或降级流程。

## 6. 三类能力的第一版示例

### 6.1 资料调研

输入：

```text
阅读 docs 目录，生成项目背景报告
```

输出：

- `outputs/research-report.md`
- 包含资料来源、关键结论、风险与后续问题。

### 6.2 本地项目助手

输入：

```text
分析当前项目结构，并建议如何添加配置模块
```

输出：

- 项目结构摘要。
- 建议方案。
- 如需修改文件，先展示确认。

### 6.3 任务自动化

输入：

```text
把 inbox 目录下的文本整理成 todo.md
```

输出：

- `outputs/todo.md`
- 按优先级、负责人、截止时间归类。

## 7. 第二版打磨方向

第二版重点不是增加页面，而是提高工程可信度：

- 权限模型：读取、写入、执行命令分级授权。
- 工具沙箱：限制可访问目录、命令白名单、超时控制。
- 异常分类：用户错误、工具错误、模型错误、系统错误分层处理。
- 重试策略：可重试错误自动重试，不可重试错误直接降级。
- 断点恢复：任务状态持久化，支持继续上次会话。
- 评测集：准备固定任务集，度量完成率、步骤数、失败原因。
- 可观察性：结构化日志、trace id、每轮 token/耗时/工具成本统计。
- 人工介入点：高风险动作前必须确认，复杂歧义主动询问。
- Prompt 版本化：记录 Planner、Reflector、Reporter 的 prompt 版本。
- Tool Contract 测试：每个工具有 schema、示例和失败用例。
- 记忆系统升级：从本地摘要检索升级为向量检索、记忆置信度和记忆冲突处理。
- 压缩策略升级：引入多级摘要、按任务阶段保留上下文、压缩质量评测。

## 8. 面试讲解重点

推荐讲解顺序：

1. 先演示 TUI 中连续对话完成一个任务。
2. 展示三层 Agent Loop：ReAct、Reflection、工程兜底重试。
3. 展示工具协议和文件写入确认。
4. 展示长期记忆和上下文压缩，说明系统如何支持持续迭代。
5. 展示执行日志，说明为什么可审计。
6. 展示第二版路线，说明如何从 demo 进化为可靠产品。

一句话总结：

> Manus Mini V1 是一个小而完整的 TUI Agent，重点证明候选人理解 Agent 不是一次 LLM 调用，而是围绕会话、目标、工具、状态、权限和反馈循环搭建的工程系统。
